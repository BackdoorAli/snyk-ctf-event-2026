# Contributing

Thank you for your interest in contributing to this repository.

## Project Nature

This repository is a **documentation-focused portfolio project** for the **Snyk CTF Event 2026**. It is intended to preserve environment setup notes, screenshots, challenge observations, and lessons learned from an authorised capture-the-flag event.

This is **not** an actively maintained software tool, framework, or exploit project. Contributions should therefore focus on improving the quality, clarity, accuracy, and professionalism of the documentation.

## Contribution Goals

Helpful contributions include:

- fixing typos, grammar, or formatting issues
- improving clarity in setup or workflow explanations
- correcting inaccurate technical descriptions
- improving screenshot references and file naming consistency
- refining repository structure and documentation flow
- suggesting better wording for portfolio presentation
- improving Markdown readability and consistency
- other challenges where people who were involved and would like to contribute on how they got there, including steps taken to achieve completion of said challenges

## Out of Scope

The following types of contributions are generally out of scope for this repository:

- adding offensive tooling
- turning the repository into an exploit kit or attack framework
- adding unsupported claims about the event or challenge outcomes
- uploading secrets, tokens, certificates, or private local files
- changing the project into a maintained software product
- submitting generated or low-quality bulk edits without review

## Before You Contribute

Before opening a pull request, please make sure that:

1. Your changes match the documentation-only purpose of the repository.
2. Sensitive information is not included anywhere in the contribution.
3. Screenshot names, folder names, and document references stay consistent.
4. The tone remains professional, direct, and portfolio-appropriate.
5. Any technical claims are accurate and supported by the repository contents.

## Recommended Workflow

For documentation contributions, use the following workflow:

1. Fork the repository.
2. Create a focused branch for your change.
3. Make small, reviewable edits.
4. Check all Markdown formatting before submission.
5. Open a pull request with a clear summary of what changed and why.

## Commit Guidance

Clear commit messages are preferred. Examples:

```text
docs: clarify Burp proxy listener setup
docs: fix screenshot references in README
docs: improve challenge timeline wording
docs: add responsible disclosure note
```

## Pull Request Expectations

A good pull request should include:

- a concise explanation of the change
- the reason the change improves the repository
- any affected files or sections
- confirmation that no sensitive data was added

Where applicable, keep pull requests narrow in scope instead of bundling unrelated edits together.

## Issues

Issues are welcome for:

- documentation corrections
- broken references
- structural cleanup suggestions
- portfolio presentation improvements
- accidental inclusion of files that should be removed

Please do not use issues for:

- requests for live challenge help
- support for attacking real systems
- requests to operationalize the repository as a tool
- disclosure of sensitive information in public

For sensitive matters, follow the guidance in `SECURITY.md`.

## Style Expectations

Contributions should follow these style expectations:

- professional and straightforward language
- no emojis
- no exaggerated claims
- concise and technically accurate wording
- consistent capitalisation of file and folder names
- consistent Markdown heading structure

## Final Note

Contributions are appreciated when they improve accuracy, clarity, and long-term readability while respecting the repository's limited, documentation-focused scope.
